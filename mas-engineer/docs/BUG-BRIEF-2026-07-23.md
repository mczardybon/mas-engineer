## Bugs mas-engineer must self-repair

### Bug 1: Translator sub-agent bypasses delegate tool

**Symptom:** translator/medium and /hard fail with "marker 'Hallo' not found".
The sub-agent uses `shell` to invoke `goose run --recipe ...` instead of
the `delegate` tool. Result: the orchestrator's output never reaches the
wrapper that emits the Hallo marker.

**Evidence (teams-25 translator-medium.log):**
```
▸ [subagent:203] shell
  command: cd /workspace/mas-engineer-src/mas-engineer && goose run --recipe
  /root/.config/goose/recipes/translator-orchestrator.yaml --params "..."
  timeout_secs: 300
```

**Root cause:** The wrapper recipe at `/tmp/e2e_teams_recipes/translator-medium.yaml`
says "You MUST invoke the translator_team sub_recipe via the delegate tool" but
the agent ignores this and shells out manually. Either the instructions are not
strong enough, OR there is no proper "must use delegate" enforcement in the
mas-engineer framework.

**Possible fix direction:** mas-engineer's recipe scaffolding should make
sub-recipes prefer the `delegate`/`load` tool by default, and ONLY allow
shell-based `goose run` when explicitly configured. Or: the wrapper should
inject a `<system-reminder>` block that re-asserts "use delegate" if the
agent starts a `shell` with `goose run` in it.

### Bug 2: Sales sub-agent infinite loops within a step

**Symptom:** sales/medium and sales/hard time out at 603s (the wrapper
timeout). The sub-agent runs python scripts that crawl 4-6 URLs each,
drains rate limits, and never emits the LEAD-DONE marker.

**Evidence (teams-25 sales-medium.log):**
The sub-agent reached Konfuzio/Merantix verification with multiple
python3 << 'PYEOF' scripts each hitting 1-3 URLs. Output cut off at
603s before any LEAD-DONE marker.

**Root cause:** The lead-scraper and lead-verifier recipes have
`max_steps: 20` and `max_steps: 30` respectively, but a single
"step" can include a python heredoc that does many curl calls.
`max_steps` does not limit nested tool calls.

**Possible fix direction:** mas-engineer should pass
`--max-tool-repetitions 2` to delegated sub-recipes so that the same
tool (e.g. `shell`) cannot be called more than 2 times in a row
without the agent having to reason. Or: the recipe schema should
add a `max_curl_calls` enforcement mechanism (currently only prompt
text "MAX_CURL_CALLS = 5" which agents ignore).

### Bug 3 (root): Verification theater still possible

**Symptom:** mas-engineer's IM-pipeline can produce commits that look
"complete" (e.g. 602648a) but only change 1 line. The metrics
("140/140 PASS 100%") were not anchored to the e2e test runner.

**Root cause:** The existing `sub_mas-pre-push-validator` runs 8
checks (YAML validity, secrets, hardcoded paths, etc.) but checks
**structure**, not **behavior**. It never runs `tools/e2e_teams.py`
or any e2e test. So a commit that adds a one-line "fix" passes the
pre-push gate even when the underlying mas-engineer framework is
broken (as 602648a demonstrated).

**IM-pipeline context (clarification by Hermes after user question):**
The IM-pipeline (FIND→RANK→DESIGN→DEVELOP→TEST) is sufficient for
*structural* code quality. It is NOT sufficient for *behavioral*
correctness of multi-agent recipes because:

- The validator reads `.state/pipeline/findings.yaml` which is
  produced by im-finder. If im-finder never analyzed the e2e
  test runner, there are no findings about e2e regressions.
- The validator cannot tell the difference between `delegate` (the
  intended pattern) and `load`/`shell` (workarounds the sub-agent
  invents). That is a semantic difference only visible by running
  the recipe and checking the sub-recipe output.
- Pre-push-validator check 1 is no-op if `.state/pipeline/findings.yaml`
  is missing, so a fresh commit that did not run im-finder passes.

**Fix direction (concrete, not architectural):** Add a 9th check to
`sub_mas-pre-push-validator` (or a new check in `.githooks/pre-push`)
that runs `python3 tools/e2e_teams.py --fast-subset` and fails the
push if pass rate is below 7/9 OR if any test regressed vs the last
recorded result. The infrastructure is already there:

- `tools/e2e_teams.py` produces `e2e-results/<timestamp>/raw-results.json`
- The latest run's `raw-results.json` can be the baseline
- A simple diff (current pass count - baseline pass count < 0) catches
  regressions; a hard floor (pass count < 7) catches absolute drops

This is a 30-50 line addition to the pre-push-validator's check
sequence, not a framework change. No new schema fields, no new
extension, no breaking change to existing recipes.

Hermes' initial framing of this as "framework-level enforcement"
was over-engineered. The actual fix is small and fits the existing
IM-pipeline.
