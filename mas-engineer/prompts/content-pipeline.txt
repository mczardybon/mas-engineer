Build a complete Multi-Agent System called "content-pipeline" at
/tmp/content-pipeline. Do NOT just plan it — actually create the
files, run live tests, and report results.

ARCHITECTURE: Hierarchical tree with manager + 3 tiers (5 agents)

  topic input
      |
      v
  pipeline-manager          (root — orchestrates, enforces gates)
      |
      +-- tier 1 (research) --------------------------+
      |                                              |
      +-> topic-researcher                            |
              (gathers facts, sources, angles)       |
              |                                       |
              +-- tier 2 (parallel creative) --------+
              |                                       |
              +-> writer-blogger                      |
              |   (writes blog draft)                |
              |                                       |
              +-> writer-social                       |
              |   (writes social posts)              |
              |                                       |
              +-> writer-newsletter                   |
                  (writes newsletter version)        |
                      |                               |
                      +-- tier 3 (gate) -----------+   |
                      |                           |   |
                      +-> editor-quality-checker  |   |
                          (one per channel, OK)   |   |
                          |                       |   |
                          v                       |   |
                      pipeline-manager <----------+   |
                      (final gate: publish)            |


STEP 1 — Initialize the project skeleton
Run: python3 ~/.config/goose/recipes/tools/dev_generic_init.py
     --init /tmp/content-pipeline --components all

STEP 2 — Create 5 agents in /tmp/content-pipeline/recipe/sub/:

1. pipeline-manager.yaml
   - Root orchestrator. Receives {topic, target_channels}
   - Dispatches to topic-researcher, then awaits 3 parallel writers
   - Enforces quality gate: every piece must pass editor check
   - Final output: {research_summary, deliverables: {...}}
   - HARD GATE: blocks publish if any editor reports severity>high

2. topic-researcher.yaml
   - Takes a topic, returns:
     {facts: [str], sources: [{url, title, credibility}],
      angles: [str], keywords: [str]}
   - TIER 1 — must complete before tier 2 starts.

3. writer-blogger.yaml  (TIER 2 — parallel)
   - Receives {topic, research}
   - Writes 800-1200 word blog draft
   - Returns: {channel: "blog", markdown, meta_description, word_count}

4. writer-social.yaml  (TIER 2 — parallel)
   - Receives {topic, research}
   - Writes 5 social posts (twitter, linkedin, mastodon) + 1 thread
   - Returns: {channel: "social", posts: [{platform, text, hashtags}]}

5. writer-newsletter.yaml  (TIER 2 — parallel)
   - Receives {topic, research}
   - Writes email newsletter (subject, preview, body, CTA)
   - Returns: {channel: "email", subject, preview, body_markdown, cta}

(All 3 writers run in PARALLEL after researcher. The orchestrator
fans them out via 3 sub_recipes and waits for all.)

STEP 3 — Create the editor layer (used as gate):
   /tmp/content-pipeline/recipe/sub/editor-quality-checker.yaml
   - Receives {piece} (one writer output)
   - Checks: factual claims vs research, tone, brand voice, length,
     clarity, grammar
   - Returns: {approved: bool, score: int, issues: [str]}

STEP 4 — Create orchestrator root recipe
/tmp/content-pipeline/recipe/content-pipeline.yaml that documents
the 3-tier tree and references all 6 sub-recipes. Manager is
sub_recipe of root; manager internally references researcher + 3
writers + editor.
